SECTION 1 — POLYMARKET ACCOUNT & API CREDENTIALS
Priority: HIGH — Q-Sigma cannot write the signing logic without this
════════════════════════════════════════════════════════════

[ ] 1.1  Create a dedicated Polymarket trading account
         URL: https://polymarket.com
         → Use a NEW account separate from any personal Polymarket account
         → Connect with a dedicated EVM wallet (see Section 2 below)

[ ] 1.2  Generate Polymarket L1 API Keys
         These are derived from your wallet signature — do it ONCE and save them.
         Method A (easiest): Use the official TypeScript client:
           npm install @polymarket/clob-client ethers
           node -e "const {ClobClient} = require('@polymarket/clob-client'); ..."
         Method B: POST https://clob.polymarket.com/auth/api-key with signed payload
         
         You will receive THREE values — save all three in a password manager:
           → POLY_API_KEY        (looks like a UUID)
           → POLY_API_SECRET     (long base64 string)
           → POLY_API_PASSPHRASE (short passphrase you set)

[ ] 1.3  Confirm your trading wallet address
         The wallet address that will be signing orders (maker address).
         → Give this to the agents so it can be hardcoded in config (not secret).
         → This is the address ALL maker orders will appear under on-chain.

[ ] 1.4  Test API connectivity (quick sanity check)
         curl https://clob.polymarket.com/
         → Should return "OK"
         → If you get blocked, you may need a VPN or the API is geo-restricted.

         Every on-chain order settlement requires gas in MATIC.
         → ~$50–$100 worth of MATIC is enough to start

[ ] 2.4  Approve USDC spending on Polymarket CTF Exchange contract
         Before the first trade can settle, you must sign an approval transaction.
         The agents will handle this programmatically, but you need to be ready

SECTION 4 — AWS ACCOUNT & SERVER
Priority: MEDIUM — Needed for Phase 5 (production). Local dev works without it.
════════════════════════════════════════════════════════════

[ ] 4.1  Create AWS account (if you don't have one)
         URL: https://aws.amazon.com
         → Enable billing alerts immediately after creation
         → Set budget alarm at $500/month to avoid surprise bills

[ ] 4.2  Provision the bare-metal server
         Recommended: AWS EC2 c7i.metal-24xl (US-East-1, N. Virginia)
         Alternative: i4i.metal (if NVMe storage is priority)
         
         IMPORTANT: This is expensive. Know the costs before provisioning:
           → On-demand: ~$5/hour ≈ $3,600/month
           → 1-Year Reserved: ~$1,800/month (pay upfront for discount)
           → DO NOT use Spot instances — they can be terminated mid-trade
         
         Required settings during launch:
           → Enable Nitro Enclaves: EC2 console → Instance Settings → Nitro Enclave → Enable
           → ENA networking enabled (should be by default on metal instances)
           → OS: Ubuntu 22.04 LTS (Ubuntu Server AMI)
           → Storage: at least 2TB NVMe (instance storage or EBS io2)

[ ] 4.3  Create SSH key pair and connect to server
         → Generate an SSH key: ssh-keygen -t ed25519 -C "blink-prod"
         → Add public key to EC2 instance during launch
         → Test: ssh -i blink-prod.pem ubuntu@YOUR_SERVER_IP
         → Save the private key securely (you will add it to GitHub Secrets)

[ ] 4.4  Set AWS IAM permissions
         The server needs IAM permissions to use Nitro Enclaves KMS.
         → Create IAM role: "blink-server-role"
         → Attach policy: AmazonEC2FullAccess + NitroEnclave KMS permissions

SECTION 6 — SPORTS DATA API (Failsafe Agent)
Priority: MEDIUM — The 3-second failsafe polls live scores during in-play
════════════════════════════════════════════════════════════

[ ] 6.1  Choose and subscribe to a live sports data provider
         The bot needs REAL-TIME score/odds updates to trigger the failsafe.
         Polymarket's own /price endpoint is used first, but a sports data
         feed gives earlier signals before the CLOB reacts.
         
         Options (in order of recommendation):
         
         Option A: The Odds API
           URL: https://the-odds-api.com
           Price: ~$100-200/month for live odds
           Coverage: All major sports
           Latency: ~1-5 second update frequency
         
         Option B: Sportradar
           URL: https://sportradar.com
           Price: Custom (expensive, enterprise)
           Coverage: Professional-grade, used by sportsbooks
           Latency: <1 second for some sports
         
         Option C: Use only Polymarket /price endpoint (FREE)
           Poll GET /price every 100ms during the 3-second window
           Lowest cost option — may be sufficient for Phase 1
         
         DECISION NEEDED: Which provider do you want?
         Recommended start: Option C (free, good enough for testing)
         
         → Provide: SPORTS_API_KEY (if using Option A or B)
         → Provide: SPORTS_API_BASE_URL

[ ] 6.2  Whitelist the bot server IP with the sports API provider
         Most sports data providers require IP whitelisting for their feeds.
         → Once server is provisioned (Section 4), provide the static IP.

SECTION 8 — LOCAL DEVELOPMENT ENVIRONMENT
Priority: HIGH — Needed before agents can start coding and testing
════════════════════════════════════════════════════════════

[ ] 8.1  Enable WSL2 on your Windows machine
         Open PowerShell as Administrator:
           wsl --install
           wsl --set-default-version 2
         → Select Ubuntu 22.04 LTS as the distro
         → Set up a username and password for WSL2

[ ] 8.2  Configure WSL2 performance (more RAM and cores)
         Create/edit C:\Users\YOUR_NAME\.wslconfig:
           [wsl2]
           memory=16GB
           processors=8
           swap=8GB
         Then restart WSL2: wsl --shutdown

[ ] 8.3  Install Rust inside WSL2
         Open Ubuntu (WSL2) terminal:
           curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh
           source $HOME/.cargo/env
           rustup toolchain install stable
           rustup toolchain install nightly
           rustup target add x86_64-unknown-linux-gnu

[ ] 8.4  Install Foundry (for smart contract testing)
         Inside WSL2:
           curl -L https://foundry.paradigm.xyz | bash
           foundryup
         → Installs: forge, cast, anvil, chisel

[ ] 8.5  Install Docker Desktop (for local ClickHouse + Reth dev)
         URL: https://docs.docker.com/desktop/windows/
         → Enable WSL2 integration in Docker Desktop settings

[ ] 8.6  Install Cursor IDE
         URL: https://cursor.sh
         → Install the rust-analyzer extension
         → Open WSL2 folder from Cursor (it will use remote mode)
         → This gives the AI full project context for coding assistance

SECTION 9 — FINANCIAL DECISIONS & RISK PARAMETERS
Priority: HIGH — These config values need YOUR approval before go-live
════════════════════════════════════════════════════════════

[ ] 9.1  Set your starting trading capital
         Default config assumes $10,000 USDC starting NAV.
         The following circuit breakers are based on this:
           → Max single order:         $1,000   (10% of $10k NAV)
           → Daily loss limit:         $200     (2% of $10k NAV)
           → VaR circuit breaker:      $500     (5% of $10k NAV)
           → Max concurrent orders:    10
         
         DECISION: What is your actual starting capital?
         → You MUST update these values in the config before going live.

[ ] 9.2  Confirm the SIZE_MULTIPLIER
         The bot copies RN1's trade size multiplied by:
           Your_Size = RN1_Size × 0.02
         
         Example: RN1 bets $50,000 → your bot bets $1,000
         This is a very conservative 2% copy ratio.
         
         DECISION: Do you want to adjust this? Range: 0.005 – 0.05 recommended.

[ ] 9.3  Confirm the failsafe threshold
         Current setting: Cancel if implied probability shifts >1.5% during 3s window
         Conservative = lower threshold = more cancellations = fewer fills
         Aggressive = higher threshold = fewer cancellations = more risk
         
         DECISION: 1.5% is default. Accept or change?
